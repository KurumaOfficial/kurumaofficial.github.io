# Prompt For Next Chat

Скопируй текст ниже в следующий чат, если нужно разнести сайт из одного HTML-файла на нормальную структуру.

```text
У меня есть статический сайт для GitHub Pages, и сейчас он собран как один большой HTML-файл. Нужно аккуратно разнести проект на нормальную структуру файлов, но без потери текущего дизайна и поведения.

Контекст:
- Проект лежит в папке `web`
- Основные файлы сейчас: `web/index.html` и `web/website.html`
- Сейчас это фактически один и тот же монолитный сайт
- Внутри HTML находятся:
  - вся разметка
  - весь CSS
  - весь JS
  - встроенные данные сайта `DEFAULT_SITE_DATA`
  - скрытая админ-панель
  - логика локального сохранения
  - логика сохранения в GitHub через token
  - логика загрузки файла товара в GitHub через Contents API
- Сайт опубликован на GitHub Pages
- Нельзя ломать существующий внешний вид, UX и текущую логику публикации

Что нужно сделать:
1. Изучить текущий `web/website.html` и понять всю структуру.
2. Разнести проект на нормальные файлы и папки.
3. Сделать архитектуру, с которой дальше реально удобно работать.
4. Сохранить совместимость с GitHub Pages.
5. Сохранить текущую админку и весь рабочий функционал.

Желаемый результат по структуре примерно такой:
- `web/index.html`
- `web/assets/css/`
- `web/assets/js/`
- `web/assets/files/`
- `web/assets/icons/` если потребуется
- `web/src/data/`
- `web/src/admin/`
- `web/src/github/`
- `web/src/components/` если это уместно

Требования:
- Не делай абстрактный рефакторинг ради рефакторинга.
- Делай прагматично: код должен стать проще поддерживать.
- Не ломай визуал.
- Не ломай скрытый вход в админку.
- Не ломай локальное сохранение.
- Не ломай общее сохранение через GitHub token.
- Не ломай загрузку файлов товара в репозиторий.
- Сохрани текущую логику предпросмотра, применения и отката черновика.
- Если понадобится, вынеси данные сайта в отдельный JSON или JS-модуль, но так, чтобы админка всё ещё могла публиковать актуальный контент через GitHub API.
- Если для GitHub Pages удобнее оставить часть данных в JS, это допустимо, но решение должно быть обоснованным.
- Если нужно, оставь `website.html` как dev-вход, а `index.html` как production-вход, либо аккуратно убери дублирование, но без поломки публикации.

Что особенно важно:
- Перед изменениями сначала опиши текущую архитектуру и план миграции.
- Потом вноси изменения поэтапно.
- После каждого этапа проверяй, что ничего не сломалось.
- Минимизируй регрессии.
- Не удаляй рабочий функционал только ради упрощения.

Что хочу получить в результате:
- Чистую файловую структуру
- Более читаемый CSS
- Более читаемый JS
- Отделение публичной части от логики админки
- Отделение GitHub sync логики от UI-логики
- Ясную точку входа для данных сайта
- Понятный README по новой структуре

Дополнительно:
- Если увидишь мёртвый код, дублирование или слабые места, убери это только если уверен, что поведение не изменится.
- Если есть спорное решение, сначала объясни trade-off.
- После рефакторинга покажи итоговую структуру файлов и кратко объясни, где теперь что лежит.
- Если сможешь, добавь простой dev workflow без тяжёлых зависимостей.

Работай прямо с файлами в проекте, а не просто описывай теорию.
```
