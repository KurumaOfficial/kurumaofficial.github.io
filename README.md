# Aleph Studio Website

Статический сайт Aleph Studio для GitHub Pages. После рефакторинга проект больше не держит весь UI, CSS, JS и данные в одном HTML-файле: публичная страница, админка, GitHub sync и контент разнесены по отдельным файлам без сборщика и без потери текущего поведения.

## Что осталось неизменным

- Внешний вид публичной страницы.
- Скрытый вход в админку: `→ → ← → ← → →`, затем клик по верхнему логотипу.
- Черновик, preview, применение и откат изменений.
- Локальное сохранение через `localStorage`.
- Публикация для всех через GitHub token.
- Загрузка файлов товара в репозиторий через GitHub Contents API.
- Совместимость с GitHub Pages и `.nojekyll`.

## Как это публикуется

В этой рабочей папке сайт сейчас живёт внутри каталога `web/`, но при публикации в `kurumaofficial.github.io` содержимое этого каталога раскладывается в корень GitHub Pages-репозитория.

## Структура в Pages-репозитории

```text
/

  docs/

  website.html
  README.md

  assets/

    components/
      public-site.js
    core/
      constants.js
      site-utils.js
    data/

    github/

```

## Где что лежит

- `index.html` — production entrypoint для GitHub Pages. Здесь осталась только HTML-разметка страницы и подключения CSS/JS.
- `website.html` — совместимый redirect на `index.html`, чтобы не поддерживать второй дубликат страницы.
- `assets/css/main.css` — стабильный CSS entrypoint, на который ссылается `index.html`. Он сохранён как один внешний путь для GitHub Pages и теперь подключает остальные стили через `@import`.
- `assets/css/public.css` — стили публичной части сайта.
- `assets/css/admin.css` — стили скрытой админки.
- `assets/files/` — публичные файлы, на которые могут ссылаться карточки товаров.
- `src/app.js` — лёгкий bootstrap, который поднимает публичную часть и админку.
- `src/components/public-site.js` — рендер публичного сайта: витрина, каталог, команда, соцссылки, reveal-анимации, glow-эффекты, toast.
- `src/admin/editor.js` — логика скрытой админки, Home-dashboard, работа с формами, черновиком, local storage и действиями редактора.
- `src/github/publisher.js` — отдельный слой публикации в GitHub: чтение/обновление файлов через Contents API и загрузка прикреплённых файлов товара.
- `src/data/site-data.js` — единственная точка входа для контента сайта. Именно этот модуль по умолчанию обновляется при публикации через админку.
- `src/core/constants.js` и `src/core/site-utils.js` — общие константы и чистые утилиты, которые используются и публичной частью, и админкой, и GitHub sync.
- `scripts/dev-server.mjs` — простой локальный сервер без зависимостей.
- `docs/ARCHITECTURE.md` — устройство проекта после рефакторинга.
- `docs/GITHUB_PAGES.md` — правила совместимости и публикации для GitHub Pages.

## Архитектура и trade-off

- Данные вынесены не в `json`, а в JS-модуль `src/data/site-data.js`.
Причина: админке нужно уметь публиковать контент обратно в репозиторий без сборки и без дополнительного рантайма. Обновление JS-модуля через GitHub API даёт простую схему: сайт импортирует данные напрямую, а админка меняет один известный файл.
- `website.html` не удалён, а превращён в redirect.
Причина: это убирает дублирование и сохраняет совместимость со старыми ссылками/привычками.
- `index.html` по-прежнему смотрит только на `assets/css/main.css`, но сами стили теперь разделены на `public.css` и `admin.css`.
Причина: это сохраняет один стабильный публичный путь для GitHub Pages и одновременно отделяет публичную часть от админки.
- `assets/icons/` не создавался.
Причина: текущие иконки и логотипы уже хранятся inline через SVG-символы в `index.html`, и отдельная папка пока не даёт выигрыша.

## Как открыть админку

1. На странице введи последовательность `→ → ← → ← → →`.
2. В течение короткого окна нажми на верхний логотип.
3. Откроется скрытая панель управления.

## Как работает сохранение

Глобальные действия теперь собраны на вкладке `Главная`: именно там находятся состояние черновика, локальное сохранение и публикация для всех.

### Локально

- Кнопка `Сохранить локально` сохраняет нормализованный контент в `localStorage`.
- Изменения видны только в текущем браузере.
- Если в черновике были прикреплённые файлы, они в GitHub не отправляются и при следующей общей публикации их нужно выбрать заново.

### Для всех

- Кнопка `Сохранить для всех` использует GitHub token с правом `Contents: Read and write`.
- По умолчанию обновляется файл `src/data/site-data.js`.
- Если у товара выбран файл, он загружается в репозиторий первым, затем обновляется контент сайта.
- Логика публикации при необходимости умеет работать и с `.json` или `.html`, но штатный путь проекта сейчас один: `src/data/site-data.js`.

Текущий GitHub Pages репозиторий по умолчанию:

```text
KurumaOfficial/kurumaofficial.github.io
```

## Локальная разработка

Сборка не нужна. Достаточно Node.js.

Запуск простого сервера из корня опубликованной раскладки:

```powershell
node scripts/dev-server.mjs
```

После запуска сайт будет доступен на `http://localhost:4173`.

Быстрая проверка модулей после изменений:

```powershell
node --experimental-default-type=module --check src/app.js
```

## Что проверять после изменений

- Открывается `index.html` и подтягивает `assets/css/main.css` и `src/app.js`.
- `assets/css/main.css` корректно импортирует `public.css` и `admin.css`.
- Секретный вход в админку по-прежнему работает.
- Вкладка `Главная` открывается первой и показывает глобальные действия по черновику и публикации.
- `Применить`, `Отменить`, `Сохранить локально` и `Сохранить для всех` работают с тем же UX, что и раньше.
- Карточка товара по-прежнему может отправить файл в репозиторий через GitHub token.
- Публичная страница рендерится из `src/data/site-data.js`.
